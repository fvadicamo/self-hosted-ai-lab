# 05 - n8n

n8n workflow automation and AI platform with PostgreSQL backend, running behind Caddy reverse proxy.

## Placeholders

| Placeholder | Description | Example |
|---|---|---|
| `<DOMAIN_N8N>` | Domain for n8n (must match Caddyfile) | `n8n.example.com` |
| `<POSTGRES_PASSWORD>` | PostgreSQL password | Generate with `openssl rand -base64 32` |
| `<N8N_ENCRYPTION_KEY>` | n8n credential encryption key | Generate with `openssl rand -hex 32` |
| `<USER>` | Admin username on the server | `deploy` |
| `<IP_ADDRESS>` | Server IPv4 | `203.0.113.10` |

## Step 1 - Create directory structure

```bash
sudo mkdir -p /srv/docker/n8n/n8n-data /srv/docker/n8n/postgres-data
```

Verify:

```bash
ls -la /srv/docker/n8n/
```

## Step 2 - Create .env file

```bash
sudo tee /srv/docker/n8n/.env > /dev/null << EOF
# PostgreSQL
POSTGRES_DB=n8n
POSTGRES_USER=n8n
POSTGRES_PASSWORD=<POSTGRES_PASSWORD>
POSTGRES_NON_ROOT_USER=n8n
POSTGRES_NON_ROOT_PASSWORD=<POSTGRES_PASSWORD>

# n8n
N8N_HOST=<DOMAIN_N8N>
N8N_ENCRYPTION_KEY=<N8N_ENCRYPTION_KEY>
EOF
```

**Generate secure values first:**

```bash
# PostgreSQL password
openssl rand -base64 32

# n8n encryption key
openssl rand -hex 32
```

> Note: do not reuse passwords between services. The .env file contains secrets - never commit it to version control. The encryption key protects all credentials stored in n8n. If you lose it, you lose access to all saved credentials. Back it up separately.

## Step 3 - Create docker-compose.yml

```bash
sudo tee /srv/docker/n8n/docker-compose.yml > /dev/null << 'EOF'
services:
  postgres:
    image: postgres:16-alpine
    container_name: n8n-postgres
    restart: unless-stopped
    environment:
      POSTGRES_DB: ${POSTGRES_DB}
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_NON_ROOT_USER: ${POSTGRES_NON_ROOT_USER}
      POSTGRES_NON_ROOT_PASSWORD: ${POSTGRES_NON_ROOT_PASSWORD}
    volumes:
      - ./postgres-data:/var/lib/postgresql/data
    networks:
      - n8n-internal
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER} -d ${POSTGRES_DB}"]
      interval: 10s
      timeout: 5s
      retries: 5

  n8n:
    image: n8nio/n8n:latest
    container_name: n8n
    restart: unless-stopped
    depends_on:
      postgres:
        condition: service_healthy
    environment:
      DB_TYPE: postgresdb
      DB_POSTGRESDB_HOST: postgres
      DB_POSTGRESDB_PORT: 5432
      DB_POSTGRESDB_DATABASE: ${POSTGRES_DB}
      DB_POSTGRESDB_USER: ${POSTGRES_USER}
      DB_POSTGRESDB_PASSWORD: ${POSTGRES_PASSWORD}
      N8N_HOST: ${N8N_HOST}
      N8N_PROTOCOL: https
      N8N_ENCRYPTION_KEY: ${N8N_ENCRYPTION_KEY}
      WEBHOOK_URL: https://${N8N_HOST}/
      N8N_SECURE_COOKIE: "true"
      N8N_PROXY_HOPS: "1"
    ports:
      - "127.0.0.1:5678:5678"
    volumes:
      - ./n8n-data:/home/node/.n8n
    networks:
      - n8n-internal
      - caddy-net

networks:
  n8n-internal:
    driver: bridge
  caddy-net:
    external: true
EOF
```

> Note: port `5678` is bound to `127.0.0.1` only (not publicly accessible). External traffic goes through Caddy on port 443. PostgreSQL is on `n8n-internal` network only, not reachable from Caddy or the outside. n8n connects to both networks: `n8n-internal` (database) and `caddy-net` (reverse proxy). `N8N_PROXY_HOPS=1` tells n8n to trust the `X-Forwarded-For` header from Caddy, so it sees the real client IP in logs and rate limiting.

## Step 4 - Start the stack

```bash
cd /srv/docker/n8n
docker compose up -d
```

Verify:

```bash
docker compose ps
```

Verify: both `n8n` and `n8n-postgres` are `running` (n8n-postgres shows `healthy`).

```bash
docker compose logs n8n --tail=20
```

Verify: n8n shows "Editor is now accessible via: ..." without errors.

## Step 5 - Verify web access

```bash
curl -I https://<DOMAIN_N8N>
```

Verify: HTTP 200 or 302 redirect to login page.

Open `https://<DOMAIN_N8N>` in your browser and complete the initial setup (create admin account).

## Step 6 - Backup access via SSH tunnel

If Caddy is down and you need direct access to n8n:

```bash
# From your local machine
ssh -L 5678:127.0.0.1:5678 <USER>@<IP_ADDRESS>

# Then open http://localhost:5678 in your browser
```

## AI capabilities

n8n includes built-in AI nodes that connect to cloud LLM providers. Once n8n is running, you can build AI-powered workflows without writing code.

### Available AI nodes

| Node | What it does |
|---|---|
| **AI Agent** | Autonomous agent that reasons, uses tools, and takes actions |
| **Text Classifier** | Categorizes text using an LLM (support tickets, sentiment, etc.) |
| **Information Extractor** | Extracts structured data from unstructured text |
| **Summarization Chain** | Summarizes documents or conversations |
| **Question & Answer Chain** | Answers questions based on provided context (RAG) |

### Connecting to LLM providers

In n8n, go to **Credentials** and add your API key for one or more providers:

- OpenAI (GPT-4, GPT-4o)
- Anthropic (Claude)
- Google (Gemini)
- Ollama (local models, if installed on the server)

Then use these credentials in any AI node.

### Example: AI Agent workflow

1. Create a new workflow
2. Add a **Chat Trigger** node (creates a chat interface)
3. Add an **AI Agent** node, connect it to the trigger
4. Configure the Agent with your LLM credential and optional tools (HTTP Request, Code, etc.)
5. Activate the workflow

You now have a custom AI agent accessible via n8n's built-in chat, Webhook, or any of n8n's 400+ integrations.

### Going further with AI

For a full local AI stack (Ollama + vector database + n8n), see n8n's official [Self-hosted AI Starter Kit](https://github.com/n8n-io/self-hosted-ai-starter-kit). It runs on top of the infrastructure you built in runbooks 01-04.

> Note: OpenClaw (runbook 06) and n8n are complementary. OpenClaw gives you a personal AI gateway with a chat interface. n8n gives you programmable AI workflows with automation. Both connect to the same cloud LLM providers.
